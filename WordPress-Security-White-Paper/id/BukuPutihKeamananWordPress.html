Buku Putih Keamanan WordPress

Ikhtisar

Dokumen ini merupakan analisis dan uraian mengenai pengembangan perangkat lunak inti WordPress dan pemrosesan keamanan terkait, juga sebagai bahan pemeriksaan atas keamanan inheren yang terbangun di dalam perangkat lunak. Para penentu kebijakan yang tengah mengevaluasi WordPress sebagai sistem manajemen konten atau framework aplikasi web harus menggunakan dokumen ini dalam analisis dan pengambilan keputusan mereka, sedangkan para pengembang harus merujuk pada dokumen ini untuk lebih memahami komponen-komponen keamanan serta praktik-praktik terbaik dalam penggunaan perangkat lunak.

Informasi dalam dokumen ini sesuai dengan rilis stabil perangkat lunak termutakhir, WordPress 4.1 saat dokumen ini diterbitkan, namun juga relevan bagi versi perangkat lunak termutakhir karena backwards compatibility merupakan fokus utama tim pengembang WordPress. Pengukuran dan perubahan keamanan tertentu akan dicatat saat ditambahkan dalam software inti pada rilis-rilis tertentu. Sangat dianjurkan untuk selalu menjalankan versi stabil termutakhir dari WordPress untuk memastikan fasilitas keamanan terbaik.

Ringkasan Eksekutif

WordPress adalah sistem manajemen konten yang dipergunakan untuk mendayai jutaan situs web, aplikasi web, dan blog. Saat ini, WordPress mendayai 23% lebih dari 10 juta situs web teratas di Internet. Kegunaan, ekstensibilitas, dan komunitas pengembangan yang matang menjadikannya pilihan yang populer dan aman bagi situs web besar maupun kecil.

Sejak kehadirannya pada tahun 2003, WordPress telah mengalami penguatan kontinu sehingga perangkat lunak intinya dapat mengenali dan menjalankan mitigasi atas ancaman-ancaman keamanan umum, termasuk daftar 10 Besar yang diidentifikasi oleh The Open Web Application Security Project (OWASP) sebagai kerentanan-kerentanan keamanan umum, yang dibahas dalam dokumen ini.

Tim Keamanan WordPress, berkolaborasi dengan Tim Kepemimpinan Inti WordPress dan didukung oleh komunitas global WordPress, bekerja untuk mengidentifikasi dan mengatasi permasalahan-permasalahan keamanan dalam perangkat lunak inti yang tersedia untuk distribusi dan instalasi di WordPress.org, serta merekomendasikan dan mendokumentasikan praktik-praktik keamanan terbaik bagi para pengembang plugin dan tema pihak ketiga.

Para pengembang dan administrator harus memberikan perhatian khusus mengenai penggunaan core API dan konfigurasi server di bawah permukaan yang kerap menjadi sumber kerentanan-kerentanan umum, termasuk memastikan bahwa semua pengguna mempergunakan password yang kuat untuk mengakses WordPress.

Ikhtisar tentang WordPress

WordPress adalah sebuah sistem manajemen konten (CMS) sumber terbuka. WordPress merupakan perangkat lunak CMS yang paling banyak dipergunakan di dunia dan mendayai 23% lebih dari 10 juta situs web1, dan diperkirakan menguasai sebesar 60% dari pangsa pasar situs web yang menggunakan CMS.

WordPress berada di bawah lisensi General Public License (GPLv2 or later) yang menyajikan empat kemerdekaan inti, dan dianggap sebagai "bill of rights" WordPress:

  1. Kemerdekaan untuk menjalankan program, untuk tujuan apapun.
  2. Kemerdekaan untuk mempelajari bagaimana program bekerja, dan memodifikasinya untuk menjalankan kerja-kerja yang Anda inginkan.
  3. Kemerdekaan untuk mendistribusikan ulang.
  4. Kemerdekaan untuk mendistribusikan salinan versi-versi termodifikasi Anda kepada orang lain.

Tim Kepemimpinan WordPress Core

Proyek WordPress berbentuk meritokrasi, dijalankan oleh sebuah tim kepemimpinan inti, dan dipimpin oleh co-creator dan lead developer-nya, Matt Mullenweg. Tim tersebut mengelola segala aspek proyek, meliputi pengembangan inti, WordPress.org, dan inisiatif- inisiatif komunitas.

Tim Kepemimpinan Inti terdiri dari Matt Mullenweg, lima lead developers, dan lebih dari selusin core developers dengan akses commit permanen. Para pengembang tersebut memiliki otoritas final atas keputusan-keputusan teknis, dan memimpin diskusi-diskusi arsitektur dan upaya-upaya implementasi.

WordPress memiliki sejumlah contributing developers. Beberapa di antara mereka adalah committers saat ini maupun terdahulu. Contributing developers tersebut adalah para kontributor WordPress veteran dan terpercaya yang terhormat di kalangan sesama pengembang. Apabila dibutuhkan, WordPress juga memiliki guest committers, individu- individu yang memiliki akses commit, kadang untuk komponen tertentu, atau dalam basis temporer atau ujicoba.

Core developers dan contributing developers utamanya memimpin pengembangan WordPress. Setiap versinya, ratusan pengembang mengontribusikan kode untuk WordPress. Para kontributor inti tersebut adalah relawan yang berkontribusi pada basis kode inti dengan cara-cara tertentu.

Siklus Rilis WordPress

Tiap-tiap siklus rilis WordPress dipimpin oleh satu atau beberapa pengembang inti WordPress. Tiap siklus rilis biasanya berjalan sekitar 4 bulan mulai dari pertemuan awal hingga peluncuran versi tersebut.

Setiap siklus rilis mengikuti pola berikut2:

  • Fase 1: Perencanaan dan penentuan kepemimpinan tim. Fase ini dijalankan di dalam ruang chat #core di Slack. Pemimpin rilis mendiskusikan fitur-fitur untuk rilis WordPress berikutnya. Para kontributor terlibat dalam diskusi tersebut. Pemimpin rilis akan mengidentifikasikan kepemimpinan tim untuk masing-masing fitur.
  • Fase 2: Kerja pengembangan dimulai. Para pemimpin tim membangun tim dan mengerjakan fitur yang ditugaskan pada mereka. Chat=chat rutin dijadwalkan untuk memastikan kerja pengembangan terus berjalan.
  • Fase 3: Beta. Versi-versi beta dirilis, dan beta-testers diminta untuk mulai melaporkan bug. Di fase ini, tak ada lagi commits berdasarkan permintaan perkembangan fitur. Para pengembang plugin dan tema pihak ketiga dihimbau untuk menguji kode mereka dengan perubahan-perubahan yang akan datang.
  • Fase 4: Release Candidate. Strings yang bisa diterjemahkan akan dibekukan pada fase ini. Kerja pengembangan ditargetkan pada regressions dan blockers saja.
  • Fase 5: Peluncuran. Versi WordPress diluncurkan dan disediakan pada Admin WordPress untuk pemutakhiran.

Penomoran Versi dan Rilis Keamanan

Versi mayor WordPress ditandai dengan dua sekuens pertama. Sebagai contoh, 3.5 adalah sebuah rilis mayor, demikian juga 3.6, 3.7, atau 4.0. Tak ada yang disebut sebagai "WordPress 3" atau "WordPress 4" dan tiap-tiap rilis mayor dirujuk dengan penomorannya, misalnya, "WordPress 3.9."

Rilis-rilis mayor bisa memperoleh penambahan fitur-fitur pengguna maupun developer API. Meskipun pada umumnya di dunia perangkat lunak, sebuah versi "mayor" berarti terputusnya backwards compatibility, WordPress berupaya keras untuk tak pernah memutuskannya. Backwards compatibility adalah salah satu filosofi terpenting dalam proyek ini, dimaksudkan untuk membuat proses pemutakhiran jauh lebih mudah baik bagi pengguna maupun pengembang.

Versi minor WordPress ditandai dengan sekuens ketiga. Versi 3.5.1 adalah sebuah versi minor, demikian pula 3.4.23. Rilis minor dimaksudkan untuk perbaikan kerentanan-kerentanan keamanan dan penanganan bug-bug kritis saja. Oleh karena versi-versi baru WordPress cukup sering dirilis — targetnya ialah tiap 4-5 bulan untuk rilis mayor, dan rilis minor diluncurkan sesuai kebutuhan — maka hanya ada rilis mayor dan minor saja.

Backwards Compatibility tiap Versi

Proyek WordPress memiliki komitmen kuat terhadap backwards compatibility. Komitmen ini berarti bahwa seluruh tema, plugin, dan kode kustom tetap berfungsi tiap kali perangkat lunak inti WordPress dimutakhirkan, dan mendorong para pemilik situs untuk menjalankan versi WordPress termutakhir yang aman pada situs mereka.

WordPress dan Keamanan

Tim Keamanan WordPress

Tim Keamanan WordPress tersusun dari sekitar 25 orang ahli termasuk lead developers dan periset keamanan — sekitar separuhnya adalah pekerja di Automattic (pembuat WordPress.com, platform hosting WordPress pertama dan terbesar), dan sebagian lainnya bekerja dalam bidang keamanan web. Tim ini berkonsultasi dengan para periset keamanan dan perusahaan-perusahaan hosting terkenal dan terpercaya3.

Tim Keamanan WordPress sering berkolaborasi dengan tim-tim keamanan lain guna menangani berbagai permasalahan dalam dependensi-dependensi umum, misalnya mengatasi kerentanan dalam PHP XML parser, yang digunakan oleh XML-RPC API yang terdapat dalam paket WordPress, pada WordPress 3.9.24. Resolusi kerentanan ini merupakan hasil dari kerjasama antara tim keamanan WordPress dan Drupal.

Resiko, Proses, dan Riwayat Keamanan WordPress

Tim Keamanan WordPress mempercayakan Responsible Disclosure dengan memperingatkan tim keamanan sesegera mungkin akan adanya kerentanan-kerentanan potensial. Kerentanan-kerentanan keamanan potensial bisa dilaporkan pada Tim Keamanan secara langsung melalui email ke alamat: security@wordpress.org5. Tim Keamanan berkomunikasi secara internal melalui sebuah milis privat, dan bekerja pada sebuah Trac privat yang terproteksi untuk penyelidikan, pengujian, dan perbaikan bug dan permasalahan keamanan.

Setiap laporan keamanan ditangani segera setelah diterima, dan tim bekerja untuk melakukan verifikasi kerentanan dan menentukan tingkat severitasnya. Jika terkonfirmasi, tim keamanan akan merencanakan sebuah patch untuk mengatasi permasalahan tersebut yang akan diterapkan sebagai commit pada rilis perangkat lunak WordPress yang akan datang, atau diterapkan sebagai rilis keamanan darurat, tergantung tingkat severitas permasalahan.

Untuk rilis keamanan darurat, sebuah himbauan akan diterbitkan oleh Tim Keamanan melalui situs WordPress News6 sebagai pengumuman rilis yang memuat perincian perubahan. Kredit atas responsible disclosure akan dimuat dalam himbauan tersebut untuk mendorong dan memperkuat responsible reporting yang kontinu kedepannya.

Para administrator perangkat lunak WordPress akan melihat notifikasi pada dasbor situs mereka untuk menjalankan pemutakhiran apabila rilis baru tersedia, dan setelah pemutakhiran manual dijalankan, pengguna akan dialihkan ke laman Tentang WordPress yang menyajikan rincian perubahan. Jika administrator mengaktifkan fitur pemutakhiran otomatis di balik layar, mereka akan menerima email sesudah pemutakhiran dijalankan.

Pemutakhiran di Balik Layar Otomatis untuk Rilis Keamanan

Mulai versi 3.7, WordPress memperkenalkan pemutakhiran otomatis di balik layar untuk semua rilis minor7, seperti 3.7.1 dan 3.7.2. Tim Keamanan WordPress bisa mengidentifikasi, memperbaiki, dan mengajukan peningkatan keamanan otomatis untuk WordPress tanpa mengharuskan pemilik situs untuk melakukan tindakan apapun di pihak mereka, dan pemutakhiran keamanan akan terinstal secara otomatis.

Ketika pemutakhiran keamanan diajukan bagi rilis stabil terkini WordPress, tim inti akan juga mengajukan pemutakhiran keamanan untuk semua rilis yang mampu menjalankan pemutakhiran di balik layar (mulai WordPress 3.7), sehingga versi WordPress terdahulu namun cukup mutakhir tersebut akan mengalami peningkatan keamanan.

Masing-masing pemilik situs bisa memilih untuk menonaktifkan pemutakhiran otomatis di balik layar melalui sebuah perubahan sederhana dalam file konfigurasi mereka, meskipun pengaktifan fungsionalitas tersebut sangat direkomendasikan oleh tim inti, demikian juga dalam hal menjalankan rilis WordPress stabil termutakhir.

2013 OWASP Top 10

The Open Web Application Security Project (OWASP) is an online community dedicated to web application security. The OWASP Top 10 list8 focuses on identifying the most serious application security risks for a broad array of organizations. The Top 10 items are selected and prioritized in combination with consensus estimates of exploitability, detectability, and impact estimates.

The following sections discuss the APIs, resources, and policies that WordPress uses to strengthen the core software and 3rd party plugins and themes against these potential risks.

A1 - Injection

There is a set of functions and APIs available in WordPress to assist developers in making sure unauthorized code cannot be injected, and help them validate and sanitize data. Best practices and documentation are available9 on how to use these APIs to protect, validate, or sanitize input and output data in HTML, URLs, HTTP headers, and when interacting with the database and filesystem. Administrators can also further restrict the types of file which can be uploaded via filters.

A2 - Broken Authentication and Session Management

WordPress core software manages user accounts and authentication and details such as the user ID, name, and password are managed on the server-side, as well as the authentication cookies. Passwords are protected in the database using standard salting and stretching techniques. Existing sessions are destroyed upon logout for versions of WordPress after 4.0.

A3 - Cross Site Scripting (XSS)

WordPress provides a range of functions which can help ensure that user-supplied data is safe10. Trusted users, that is administrators and editors on a single WordPress installation, and site administrators only in WordPress Multisite, can post unfiltered HTML or JavaScript as they need to, such as inside a post or page. Untrusted users and user-submitted content is filtered by default to remove dangerous entities, using the KSES library through the wp_kses function.

As an example, the WordPress core team noticed before the release of WordPress 2.3 that the function the_search_query() was being misused by most theme authors, who were not escaping the function's output for use in HTML. In a very rare case of slightly breaking backward compatibility, the function's output was changed in WordPress 2.3 to be pre-escaped.

A4 - Insecure Direct Object Reference

WordPress often provides direct object reference, such as unique numeric identifiers of user accounts or content available in the URL or form fields. While these identifiers disclose direct system information, WordPress' rich permissions and access control system prevent unauthorized requests.

A5 - Security Misconfiguration

The majority of the WordPress security configuration operations are limited to a single authorized administrator. Default settings for WordPress are continually evaluated at the core team level, and the WordPress core team provides documentation and best practices to tighten security for server configuration for running a WordPress site11.

A6 - Sensitive Data Exposure

WordPress user account passwords are salted and hashed based on the Portable PHP Password Hashing Framework12. WordPress' permission system is used to control access to private information such an registered users' PII, commenters' email addresses, privately published content, etc. In WordPress 3.7, a password strength meter was included in the core software providing additional information to users setting their passwords and hints on increasing strength. WordPress also has an optional configuration setting for requiring HTTPS.

A7 - Missing Function Level Access Control

WordPress checks for proper authorization and permissions for any function level access requests prior to the action being executed. Access or visualization of administrative URLs, menus, and pages without proper authentication is tightly integrated with the authentication system to prevent access from unauthorized users.

A8 - Cross Site Request Forgery (CSRF)

WordPress uses cryptographic tokens, called nonces13, to validate intent of action requests from authorized users to protect against potential CSRF threats. WordPress provides an API for the generation of these tokens to create and verify unique and temporary tokens, and the token is limited to a specific user, a specific action, a specific object, and a specific time period, which can be added to forms and URLs as needed. Additionally, all nonces are invalidated upon logout.

A9 - Using Components with Known Vulnerabilities

The WordPress core team closely monitors the few included libraries and frameworks WordPress integrates with for core functionality. In the past the core team has made contributions to several third-party components to make them more secure, such as the update to fix a cross-site vulnerability in TinyMCE in WordPress 3.5.214.

If necessary, the core team may decide to fork or replace critical external components, such as when the SWFUpload library was officially replaced by the Plupload in 3.5.2, and a secure fork of SWFUpload was made available by the security team15 for those plugins who continued to use SWFUpload in the short-term.

A10 - Unvalidated Redirects and Forwards

WordPress' internal access control and authentication system will protect against attempts to direct users to unwanted destinations or automatic redirects. This functionality is also made available to plugin developers via an API, wp_safe_redirect()16.

Further Security Risks and Concerns

XXE (XML eXternal Entity) processing attacks

When processing XML, WordPress disables the loading of custom XML entities to prevent both External Entity and Entity Expansion attacks. Beyond PHP's core functionality, WordPress does not provide additional secure XML processing API for plugin authors.

SSRF (Server Side Request Forgery) Attacks

HTTP requests issued by WordPress are filtered to prevent access to loopback and private IP addresses. Additionally, access is only allowed to certain standard HTTP ports.

Keamanan Plugin dan Tema WordPress

Tema Bawaan

WordPress requires a theme to be enabled to render content visible on the frontend. The default theme which ships with core WordPress (currently "Twenty Fifteen") has been vigorously reviewed and tested for security reasons by both the team of theme developers plus the core development team.

The default theme can serve as a starting point for custom theme development, and site developers can create a child theme which includes some customization but falls back on the default theme for most functionality and security. The default theme can be easily removed by an administrator if not needed.

Repositori Tema dan Plugin WordPress.org

There are approximately 30,000+ plugins and 2,000+ themes listed on the WordPress.org site. These themes and plugins are submitted for inclusion and are manually reviewed by volunteers before making them available on the repository.

Inclusion of plugins and themes in the repository is not a guarantee that they are free from security vulnerabilities. Guidelines are provided for plugin authors to consult prior to submission for inclusion in the repository17, and extensive documentation about how to do WordPress theme development18 is provided on the WordPress.org site.

Each plugin and theme has the ability to be continually developed by the plugin or theme owner, and any subsequent fixes or feature development can be uploaded to the repository and made available to users with that plugin or theme installed with a description of that change. Site administrators are notified of plugins which need to be updated via their administration dashboard.

When a plugin vulnerability is discovered by the WordPress Security Team, they contact the plugin author and work together to fix and release a secure version of the plugin. If there is a lack of response from the plugin author or if the vulnerability is severe, the plugin/theme is pulled from the public directory, and in some cases, fixed and updated directly by the Security Team.

Tim Pengulas Tema

The Theme Review Team is a group of volunteers, led by key and established members of the WordPress community, who review and approve themes submitted to be included in the official WordPress Theme directory. The Theme Review Team maintains the official Theme Review Guidelines19, the Theme Unit Test Data20, and the Theme Check Plugin21, and attempts to engage and educate the WordPress Theme developer community regarding development best practices. Inclusion in the group is moderated by core committers of the WordPress development team.

Peranan Penyedia Layanan Hosting dalam Keamanan WordPress

WordPress can be installed on a multitude of platforms. Though WordPress core software provides many provisions for operating a secure web application, which were covered in this document, the configuration of the operating system and the underlying web server hosting the software is equally important to keep the WordPress applications secure.

Catatan tentang WordPress.com dan Keamanan WordPress

WordPress.com is the largest WordPress installation in the world, and is owned and managed by Automattic, Inc., which was founded by Matt Mullenweg, the WordPress project co-creator. WordPress.com runs on the core WordPress software, and has its own security processes, risks, and solutions22. This document refers to security regarding the self-hosted, downloadable open source WordPress software available from WordPress.org and installable on any server in the world.

Apendiks

Core WordPress APIs

The WordPress Core Application Programming Interface (API) is comprised of several individual APIs23, each one covering the functions involved in, and use of, a given set of functionality. Together, these form the project interface which allows plugins and themes to interact with, alter, and extend WordPress core functionality safely and securely.

While each WordPress API provides best practices and standardized ways to interact with and extend WordPress core software, the following WordPress APIs are the most pertinent to enforcing and hardening WordPress security:

Database API

The Database API24, added in WordPress 0.71, provides the correct method for accessing data as named values which are stored in the database layer.

Filesystem API

The Filesystem API25, added in WordPress 2.626, was originally created for WordPress' own automatic updates feature. The Filesystem API abstracts out the functionality needed for reading and writing local files to the filesystem to be done securely, on a variety of host types.

It does this through the WP_Filesystem_Base class, and several subclasses which implement different ways of connecting to the local filesystem, depending on individual host support. Any theme or plugin that needs to write files locally should do so using the WP_Filesystem family of classes.

HTTP API

The HTTP API27, added in WordPress 2.728 and extended further in WordPress 2.8, standardizes the HTTP requests for WordPress. The API handles cookies, gzip encoding and decoding, chunk decoding (if HTTP 1.1), and various other HTTP protocol implementations. The API standardizes requests, tests each method prior to sending, and, based on your server configuration, uses the appropriate method to make the request.

Permissions and current user API

The permissions and current user API29 is a set of functions which will help verify the current user's permissions and authority to perform any task or operation being requested, and can protect further against unauthorized users accessing or performing functions beyond their permitted capabilities.
